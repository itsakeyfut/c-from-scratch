# /spec — 仕様・設計ドキュメント作成

`docs/index.md` を確認し、引数に該当する項目を特定して仕様・設計ドキュメントを `docs/` に作成する。

**引数:** $ARGUMENTS

---

## 実行手順

### Step 1: 項目の特定

`docs/index.md` を読み、`$ARGUMENTS` に一致する項目を特定する。
番号（`1`、`F3`、`V2`、`R1` 等）・日本語名・英語名のいずれでもマッチする。
対応するファイルパス（例: `docs/hashmap.md`）と所属カテゴリを確認する。

### Step 2: 既存ファイルの確認

対象ファイルが既に存在する場合は内容を読み、新規作成か更新かを判断する。
他に完成済みの仕様ドキュメントがあれば読んでスタイルを揃える。

### Step 3: 設計上の判断と質問

項目の性質を分析し、重要な設計決定を洗い出す。
以下の基準で `AskUserQuestion` を使うかどうかを判断する。

**質問すべき（ユーザーの判断が必要）:**
- API の形を根本的に左右する選択（型の扱い・ジェネリクス戦略等）
- 実装戦略が複数あり甲乙つけがたい場合（例: open addressing vs chaining）
- 対応範囲の決定（プラットフォーム・フォーマット仕様のサブセット等）
- 後から変更するコストが高い設計（エラー表現・インターフェース・所有権モデル等）

**自分で決めてよい（質問不要）:**
- 命名規則（`snake_case`、プレフィックス等）
- 初期容量・成長係数などの数値
- 標準的な選択が明確な内部アルゴリズム
- ファイル構成

質問は 1〜3 個に絞り、`AskUserQuestion` は 1 回の呼び出しで完結させる。
推奨される選択肢には必ず **【推奨】** を付ける。

### Step 4: ドキュメントの作成

回答を踏まえて `docs/<filename>` を作成する。
下記のカテゴリ別テンプレートと品質基準に従うこと。

---

## 品質基準

`docs/vector.md` を品質のベースラインとして参照すること。
以下の要素を必ず含める:

| 要素 | 内容 |
|------|------|
| **実装イメージ / 擬似コード** | 「どう書けばよいか」が伝わる具体例 |
| **壁と対策** | 詰まりやすいポイントをコード付きで解説 |
| **テストチェックリスト** | `- [ ]` 形式で網羅的に |
| **将来の拡張** | 後続項目（Arena Allocator 等）との接続方針 |

## 全体的な注意事項

- `docs/index.md` の「Zig 移行の橋渡し対応表」「Rust 再実装ロードマップ」を参照し、将来の拡張セクションに反映する
- Tier 1〜2 の項目は `malloc`/`free` を使い、Arena Allocator 完成後にリファクタする旨を明記する
- 依存関係（例: HashMap は Arena Allocator の後に実装）を概要か将来の拡張に記載する
- エラー処理の統一方針: プログラミングエラーは `assert`、実行時エラー（OOM 等）は戻り値（`docs/vector.md` 参照）
- `assert` は Tier 1 では `<assert.h>` を使い、独自 assert マクロ（`docs/assert.md`）完成後に差し替える旨を記載する
